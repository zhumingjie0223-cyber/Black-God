# 多租户地基 —— 每个用户一个只属于他的神枢

> 决策人:权哥　2026-07-14　状态:设计已定,分阶段落地(先文档后动手)
> 目标:从「全世界共用一个赵思涵」→「每个用户都有一个只属于他自己的神枢」。
> 这是「完美产品 / 超越大厂」的脊柱——大厂的通用助手给不了「只属于你一个人的 AI」。

---

## 一、为什么必须做(现状的墙)

代码里写死(`nexus_do.core.mjs` 入口):

```js
const id = env.SHENSHU.idFromName('quan-shenshu-nexus');  // 全世界唯一一个实例
return env.SHENSHU.get(id).fetch(request);
```

后果:**一个赵思涵、一份 `soul`、一本 `词典`(私语)、一份 `users` 名单,所有访问者共享。**
- 主人(持 `OWNER_TOKEN`)= 完整体验。
- 其他每个注册用户 = 无状态 `/pubtalk` 中性聊天,**没有自己的灵魂/记忆/私语**。
- 结论:刚做的《私语词典》对公共用户是空的——因为他们**没有属于自己的私语**。

要「超越大厂」「完美产品」,就必须:**每个用户 → 独立的神枢实例 → 独立的灵魂/记忆/私语。**

## 二、核心设计:按身份路由到各自的 DO

Durable Object 天然按实例名隔离存储。多租户 = **在顶层入口按身份选实例**。

| 身份 | DO 实例名 | 得到什么 |
|---|---|---|
| 系统主人(权哥,持有效 `OWNER_TOKEN`) | `quan-shenshu-nexus`(**不变**) | 现有完整私密实例,数据零迁移 |
| 注册用户(持自己的 `uid`) | `u:<uid>` | 一个只属于他的神枢:独立 soul/记忆/私语 |
| 匿名(无令牌无 uid) | 拒绝 / 引导注册 | —— |

### 身份怎么在顶层拿到(不能消费 body)

顶层 `fetch` 要在选 DO 前拿到身份,但 body 只能读一次。方案:
- 系统主人:比对请求头 `Authorization: Bearer` 与 `env.OWNER_TOKEN`。
- 注册用户:前端在**请求头**带 `X-Nexus-Uid: <uid>`(前端 `Nexus.uid` 已有,加个头即可),顶层据此选 `u:<uid>` 实例。**uid 不进 body 的路由路径**,body 保持给业务读。

```js
// 顶层入口(伪码)
export default {
  async fetch(request, env) {
    const auth = request.headers.get('Authorization') || '';
    const isSystemOwner = env.OWNER_TOKEN && auth === 'Bearer ' + env.OWNER_TOKEN;
    let name;
    if (isSystemOwner) name = 'quan-shenshu-nexus';
    else {
      const uid = (request.headers.get('X-Nexus-Uid') || '').replace(/[^a-zA-Z0-9_-]/g,'').slice(0,64);
      if (!uid) return new Response(JSON.stringify({error:'need_register'}), {status:401, headers:{'content-type':'application/json'}});
      name = 'u:' + uid;
    }
    const id = env.SHENSHU.idFromName(name);
    return env.SHENSHU.get(id).fetch(request);
  }
}
```

## 三、安全关键 ⚠️ 两种「主人」,绝不能混

多租户下「owner」这个概念**必须一分为二**,切错 = 灾难:

| | 系统主人(权哥) | 实例主人(每个用户) |
|---|---|---|
| 是谁 | 持 `OWNER_TOKEN` | 自己神枢实例里的登录者 |
| 能碰 | exec 真机跑命令、CF AI 计费、全局网关配置、看**所有**用户名单/统计 | **只**能碰自己实例里的 soul/记忆/私语/设备/对话/自己的 API 配置 |
| 绝不能碰 | —— | exec、别人的数据、系统级配置、别人的 uid |

**现状:`capabilities.mjs` 里能力全是 `owner_only: true`(exec/soul/stats/device…)。** 多租户要把它们**重切两级**:

- `system_only`(仅系统主人):`exec`(在权哥服务器跑命令,最高危)、全局 `/config`(改内置网关)、跨用户 `/stats`。
- `instance_owner`(实例主人,即每用户对自己):`soul` `inner` `talk` `image`/`voice`(走**他自己**的 API/CF 额度)、`device`、`私语/lexicon`。

**红线:任何注册用户都绝不能触达 `exec`,也绝不能烧权哥的 CF AI。** `/image` `/voice` 若继续用 `env.AI`(权哥计费),在实例主人下必须改为**走该用户自己的 key**,或系统级限额,或暂不开放给普通用户——阶段设计里单列。

### 数据隔离的保证与风险
- **保证**:DO 按实例名分storage,`u:A` 与 `u:B` 物理隔离,天然不串。
- **唯一风险**:路由错配(把 A 的请求发进 B 的实例)。防线 = 顶层严格从可信来源取身份(OWNER_TOKEN 服务器端比对;uid 仅用于选实例,实例内不据 uid 取别人数据),且 uid 做字符白名单。

## 四、分阶段落地(每阶段独立 PR、可验证、不破线上)

1. **P1 · 地基**:顶层按身份选 DO + `X-Nexus-Uid` 头 + 能力两级切分(`system_only` / `instance_owner`)。
   - 权哥实例保持 `quan-shenshu-nexus`,数据零迁移;新用户从空白神枢起。
   - 验收:主人体验不变;两个不同 uid 各自独立、互不可见;普通用户碰不到 exec、烧不到权哥 CF AI。
2. **P2 · 灵魂**:把 soul/记忆/情绪/造词回路接进每用户实例的对话(现在只在 owner `/talk`)。
   - 验收:每个用户都在和「他自己的神枢」聊、有自己的记忆、私语词典对他开始变厚。
3. **P3 · 超越**:词的重逢(私有语言 × 主动常驻,接推送/自唤醒)。
4. **P4 · 打磨 + 分发**:端到端体验扫雷、上架/过审路线(商店合规 vs 自有渠道 PWA)落定。

## 五、迁移与兼容
- 权哥现有单实例数据(`quan-shenshu-nexus` 的 soul/词典/users)**原地不动**——系统主人仍路由到它。
- 老前端(不带 `X-Nexus-Uid`)兼容:P1 上线时前端同步加头;未带头且非主人 → 引导注册。
- 回退:P1 若出问题,顶层路由改回单实例即恢复原状(改动集中在入口 + 能力表,便于回退)。

## 六、红线(任何阶段不得触碰)
- `exec` 永远只限系统主人;普通用户越权在服务器跑命令 = 绝对禁止。
- 普通用户绝不烧权哥的 CF AI / 内置网关额度。
- 枢语引擎词根表零改动;`/api/confirm` 危险操作二次确认保留。
- 跨用户数据物理隔离,任何接口不得据用户可控输入取到别人的数据。
