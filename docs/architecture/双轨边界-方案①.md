# 神枢双轨边界划分 —— 方案①:CF 边缘为主 · Python 降级为重活外包

> 决策人:权哥　决策日期:2026-07-13　状态:已拍板,分阶段落地
> 本文是"先出文档再动手"的那份文档,后续所有双轨相关改动以本文为准。

---

## 一、决策(方案①)

两轨不再并列当大脑,改成**主从关系**:

1. **CF 主轨(`web/nexus-do/`,Cloudflare Durable Object)= 常驻大脑主轨**
   日常对话 / 情绪 / 记忆 / 设备识别 / 永在线保活,全归它。
   这是神枢"一直在、绝不断链"灵魂的载体——边缘常驻、全球低延迟、Durable Object 持久状态。

2. **Python `server/` = 按需调用的重算力后端(降级)**
   只保留真正吃算力 / 需本地环境的重活:docker 隔离沙箱、深度多 agent 编排、本地大模型、
   code_engine 代码执行。由 CF 在需要时**远程调用**,不再独立当大脑。

3. **调用协议**:CF → Python 走一个 HTTP 端点 + 权限校验(Bearer token)。
   两轨重叠的对话 / 情绪 / 记忆逻辑,从 Python 侧标记为 `deprecated`,统一走 nexus-do。

---

## 二、为什么是方案①(而不是保留双轨或反向)

| 维度 | 方案①(CF 主轨) | 保留双轨 | Python 当主轨 |
|---|---|---|---|
| "绝不断链"的灵魂 | ✅ 边缘常驻,DO 持久,天然永在线 | ⚠️ 两套保活各自为政,易分叉 | ❌ VPS 单点,宕机即断链 |
| 全球延迟 | ✅ 边缘就近 | ⚠️ 取决于走哪轨 | ❌ 单地域 |
| 记忆 / 状态一致性 | ✅ 单一真相源(DO storage) | ❌ 两份记忆易漂移(已实测分叉) | ⚠️ 集中但离用户远 |
| 重活能力(docker/多agent/本地模型) | 靠远程调用 Python 补 | ✅ 本地直跑 | ✅ 本地直跑 |
| 运维成本 | ✅ 无常驻服务器,按需起 Python | ❌ 两套都要养 | ❌ 常年养 VPS |
| 与现有代码契合度 | ✅ `execRemote()`/`exec_brain.py` 已是这个形状 | — | ❌ 要把大脑搬离边缘 |

结论:方案① 让"常驻灵魂"落在最适合常驻的边缘,把"重活"甩给最适合跑重活的本地——各就各位。
且**现有代码已经朝这个方向长了**(见第四节),方案① 是顺势 formalize,不是推倒重来。

## 四、CF → Python 调用协议

**关键事实:这套协议不是新发明的,现有代码已经跑着它的雏形。**

nexus-do 的 `execRemote(cmd)`(`nexus_do.core.mjs`)已经在干:

```
CF Worker  ──HTTPS POST──▶  {NEXUS_EXEC_URL}/exec
  headers: Authorization: Bearer {NEXUS_EXEC_TOKEN}
  body:    { "cmd": "...", "timeout": 60 }
                                    │
                                    ▼
                          Python exec_brain.py
  ◀──JSON──  { "ok": true, "code": 0, "stdout": "...", "stderr": "..." }
```

且已内建一条红线(`nexus_do.core.mjs` 第 760 行):
**未配 `NEXUS_EXEC_URL` 就如实告知"执行脑未接入",绝不假装能跑。** 这条方案① 下继续保留。

### 4.1 协议规范(formalize + 扩展)

- **传输**:HTTPS,`POST`,`Content-Type: application/json`
- **端点**:`{NEXUS_EXEC_URL}/<capability>`
- **权限校验**:`Authorization: Bearer {NEXUS_EXEC_TOKEN}`;Python 侧设了 token 就强制校验,401 拒绝
- **配置来源**:优先 DO storage 里 App 内配置(`exec_url`/`exec_token`),回落到环境变量 `NEXUS_EXEC_URL`/`NEXUS_EXEC_TOKEN`

### 4.2 端点清单

| 端点 | 状态 | 用途 | 请求体 |
|---|---|---|---|
| `POST /health` | 已有 | 探活 | — |
| `POST /exec` | 已有 | 跑 shell | `{cmd, timeout}` |
| `POST /read` | 已有 | 读文件 | `{path}` |
| `POST /write` | 已有 | 写文件 | `{path, text}` |
| `POST /heavy/code` | 待建 | 代码生成→调试闭环 | `{task, lang, context}` |
| `POST /heavy/sandbox` | 待建 | docker 隔离执行 | `{code, lang, image?, timeout}` |
| `POST /heavy/multi-agent` | 待建 | 深度多 agent | `{goal, budget?, agents?}` |
| `POST /heavy/voice` | 待建 | 本地 TTS 合成 | `{text, voice?, emotion?}` |
| `POST /heavy/media` | 待建 | 生图/生视频 | `{prompt, kind, opts?}` |

### 4.3 统一响应约定

```json
{ "ok": true, "code": 0, "data": {…}, "stdout": "…", "stderr": "…", "error": null }
```
- `ok=false` + `note` 表示"后端未接入/拒绝",CF 主轨据此如实告诉用户,不编造。
- 流式重活(多 agent、代码调试)用 SSE:`Content-Type: text/event-stream`。

### 4.4 降级语义(灵魂不断链的关键)

**Python 后端不可达时,CF 主轨照常对话——只是重活能力暂缺,并如实说明。**
灵魂(对话/情绪/记忆/保活)全在 CF,不依赖 Python 存活;Python 只是"手和重算力",断了不影响"在不在"。

## 五、落地步骤(分阶段,绝不一次性删)

遵守"先备份/归档再删除"铁律,分四阶段:

1. **阶段一(本 PR)**:出本文档 + 给 A 类 13 个 Python 模块加 `deprecated` 头注
   (指明"方案① 下由 nexus-do 承接,勿再当主大脑扩展"),**不删任何代码**,可随时回退。
2. **阶段二**:CF 侧按 4.2 补齐 B 类重活的调用封装(`/heavy/*`),Python 侧对应实现路由。
3. **阶段三**:确认 CF 主轨功能与 A 类对等后,把 A 类 Python 代码**归档**(移入 `archive/`,非删除)。
4. **阶段四**:部署时把 `NEXUS_EXEC_URL` 指向 Python 重活后端,按需拉起(不常驻)。
