# Black God 回溯整理版（从原点到现在）

> 这份是给权哥自己的“线索总表”。
> 目标：不再被聊天记录带乱，只看这份就知道项目从哪来、现在是什么、下一步该做什么。

---

## 1. 你真正反复说的目标

你从头到尾不是在做一个普通聊天 App，你要的是：

- **一个有能力的前端 Agent**
- **Agent 本身强，不是后端强**
- **不是接口，不是写进去的文案，而是真正有实力**
- **对外定位：Black God 匹配最适合你的个人生活工作助理**
- **内部理念：黑 = 手段/技术/能力；神 = 无所不能**
- **越用越懂你**，经用户同意后分析习惯，往偏好方向发展
- **逆向 Minis，做得比它更强、更全**
- **真沙箱、真技能、自进化、缺口就新增技能补上**
- **端到端同步**，和权哥 iOS 手机尽量一致

这就是项目的原点。

## 2. 回溯时间线：项目是怎么长出来的

### 2.1 原点阶段：想要自己的 Minis
最早目标不是 App，而是：
- 想要一个自己的、可执行的、不会只聊天的 AI
- 你一开始就在找：沙箱、技能、记忆、模型调配、多模型路由
- 方向已经定死：**不是空壳，不是纯聊天，而是能自己做事的 Agent**

### 2.2 服务器能力阶段：Black-Shen / 统一版
先把能力放到服务器上：
- Shell、Python、文件、搜索、浏览器、SQL、Git、记忆、APK 等工具
- 形成了最早的可执行能力
- 后来统一成 Black God 品牌
- 80 端口那条线长期承载“统一版”

这一阶段的特点：
- **能用，但还不够像产品**
- 工具堆出来了，但 Agent 的“自我推进感”还不够强

### 2.3 逆向阶段：对标 Minis / Manus / Claude
你开始明确对标：
- **Minis**：技能系统、沙箱、Apple 原生能力
- **Manus**：规划与多步执行
- **Claude**：真模型、工具调用、服从性
- **Hermes**：记忆与上下文

这一步的结论是：
- 单靠聊天不行
- 单靠后端接口也不行
- 必须有 **Agent 核心循环 + 技能系统 + 记忆系统 + 真沙箱**

### 2.4 产品化阶段：Black God Agent OS（8765）
后来你把主线推进到 8765：
- 首页、对话、任务中心、沙箱终端、手机指挥、技能记忆
- 可视化节点
- 对话自动压缩
- 文件上传
- 人格设定
- 定时任务
- App 商城初版
- 能力海报

这一阶段才开始像一个真正的 **Agent OS 控制台**。

### 2.5 收口阶段：只留一条主线
你后来明确要求：
- 老项目要收口
- 只能保留一套主线
- API 全部指向新系统前端
- 老源码不要污染新源码

现在已经做了第一阶段收口：
- 80 已改指向 8765
- 8084 / 8086 / 8087 / 8090 的主线使用已经停掉
- 主线只剩新版 Agent OS

## 4. 你真正想要的产品口径

你想把 Black God 做成：

> **一个“匹配最适合你的个人生活工作助理”的产品化 App / Web App，而不是一个技术展示站。**

对外表达要简单：
- 它能帮人做事
- 它会记住偏好
- 它越用越懂人
- 它不是只会聊天，而是会执行

这才是你要的“产品口径”。

## 6. 现在最该做什么

### P0：收口
- 只保留一套主线
- 老项目确认后清理干净
- 不让旧源码污染新源码
- 文档里的敏感内容不要散落

### P1：打稳核心闭环
- 单 Agent 可靠性
- 任务断点续跑
- 真沙箱隔离
- 20-50 条任务测试集

### P2：产品化
- App 首页更清楚
- 首次使用更容易懂
- 结果可复制、可分享
- 中英文 / 海外场景适配

### P3：以后再说
- 多 Agent 作为增强
- iOS / Android 正式分发
- 更大的能力市场

## 7. 你这一路最关键的原则

### 原则 1：只增不减
- 现有技能不乱删
- 有缺口就新增技能补齐
- 不覆盖、不精简、不乱动已有能力

### 原则 2：先 Agent，后后端
- Agent 强，产品才强
- 后端只是支撑
- 对外展示的是“能做事”，不是“有接口”

### 原则 3：围绕真实任务，不要刻意演人群
- 针对真实场景、真实任务、真实工作流
- 不是硬凹用户画像剧情
- 产品针对性要有，但不能刻意安排

**Black God 的核心目标从来没变：**

> 做一个强 Agent，做一个会执行、会记忆、会自进化、越用越懂你的个人生活工作助理。

乱的只是过程，不是目标。
