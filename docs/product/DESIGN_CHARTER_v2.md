# Black God — 项目设计纲领（Design Charter）

> 这份文档不是 Git 流水账，而是回答四个问题：Black God 为什么存在、要解决什么、核心设计是什么、怎么一步步长成"一个真正懂你工作生活的助理"。
>
> 版本：v2（在原《核心出发点与设计由来》基础上重构、去重、补全工程化与安全边界）

---

## 0. 一句话定位（TL;DR）

> **你要的不是别人的 AI，不是一次性聊天，也不是传统问答工具，而是一个真正属于你的、主动型的工作生活助理。**
>
> 它首次安装就能在本地认识你、默认不上传云端；内部靠本地记忆、缓存、工具和分级模型调用来节约 Key；内置强沙箱负责真实执行；可配置远端承担重任务；手机 App 商城把真实 App 接进端到端任务链；技能系统从真实任务里自己长出能力、拒绝垃圾堆积——**越用越懂你，越用越强。**

它的终点不是一个网页或一个 iOS App，而是：**一个属于你的个人 Agent 操作系统（Agent OS）。**

---

## 1. 核心定位

Black God 的核心**不是炫工具，不是聊天机器人，也不是模型有多强**，而是：

> **它能不能通过本地记忆和行为画像，越来越懂你的工作和生活。**

它要懂的不是抽象的"用户"，而是你这个人：你怎么说话、工作里常做什么、生活里在意什么、有哪些项目 / 服务器 / 仓库、哪些能碰哪些不能碰、你喜欢快和直接、你希望它主动接住上下文而不是每次重新解释。

### 1.1 不是问答，是主动型助理

传统 AI 是"你问一句、我答一句"。Black God 更像一个真正的助理：

```text
你给一个目标
  → 它自己理解背景
  → 内部先查记忆和缓存
  → 能不用模型就不用模型
  → 需要模型时再调用
  → 能本地做就本地做，需要远端再上远端
  → 最后给你结果，而不是把问题丢回来
```

这套循环的本质就是 **Agent Loop**：理解需求 → 拆任务 → 选工具 → 执行 → 观察结果 → 修正 → 交付 → 沉淀经验。这是项目第一天就定下的内核，不是后来加的。

### 1.2 "不烧 Key"的成本模型

Black God 的智能不是只靠模型，而是靠一条**分级降本链路**，让大部分请求根本不到达昂贵模型：

| 层级 | 命中场景 | 成本 |
|---|---|---|
| 本地记忆 | 画像、历史任务、项目记录里已有答案 | ~0 |
| 缓存 | 相同 / 相似问题、已生成内容可复用 | ~0 |
| 规则 / 工具 | 计算、文件操作、路由查询、状态检查 | 本地算力 |
| 摘要压缩 | 长上下文先本地压缩再给模型 | 省 token |
| 轻模型 | 小事用本地 / 小模型 | 低 |
| 强模型 | 只有大事才上 Claude / GPT 级 | 高 |
| 结果沉淀 | 每次完成沉淀为记忆 / 技能 / 模板，下次更省 | 负成本（越用越省） |

> 衡量指标：**单位任务的模型调用成本应随使用时间下降**——这是这套设计成不成立的硬标准（见 §10）。

所以智能 = **本地记忆 + 缓存 + 工具 + 沙箱 + 远端配置 + 模型网关**共同组成的助理系统，而不是"一个更强的模型"。

## 2. 为什么存在（设计动机）

现有 AI 很强，但它们**都不是你的，也不够懂你**。Black God 的存在由七个一以贯之的理由推出来：

1. **现有 AI 都不属于你**：账号、模型、规则、能力边界、记忆都在别人手里，一封号 / 一限流 / 一改规则你就被动。→ 要一个真正属于自己的 AI 中枢，而不是借用别人产品里的一个聊天窗口。
2. **普通 AI 只会答、不会做**：你要的不是建议，是结果。→ 从第一天就设计成 Agent（能调工具、写代码、跑命令、改文件、部署、查网页），而不是 Chatbot。
3. **你要的是长期工作脑，不是一次性问答**：你的需求是连续的（今天做项目、明天接着做、下次自动记得哪个服务器不能碰）。→ 必须有记忆系统。
4. **单一模型不可靠**：Claude 可能拒绝、OpenAI 可能限流、DeepSeek 可能慢、Key 可能失效。→ 必须有模型网关，上层永远叫 Black God，底层自动换脑。
5. **iOS / App Store 有限制**：你要马上能用、马上能迭代。→ 先走 PWA，绕开审核与等待。
6. **既要能公开展示，又要保留真正自用版**：只做公开版不够懂你，只做私人版不能对外。→ 必须双轨。
7. **最终目标不是 App，是 Agent OS**：一个能长期进化、能替你做事、能跨设备接入的个人智能系统。

## 3. 竞品定位与差异化（它跟现成的有什么不一样）

| 维度 | ChatGPT / Claude App | Manus / Devin 等自主 Agent | **Black God** |
|---|---|---|---|
| 归属 | 厂商账号 | 厂商 SaaS | **你的设备 + 你的服务器** |
| 模型 | 绑定单家 | 绑定单家 | **模型网关，自动路由多家 / 本地** |
| 记忆 | 平台侧、有限 | 平台侧 | **本地优先、分层、可压缩、归你** |
| 执行 | 受限 / 沙箱黑盒 | 云端黑盒沙箱 | **本地 + 可配置远端 + 透明可审计** |
| 隐私 | 上云 | 上云 | **默认不上传，授权多少懂多少** |
| 形态 | 聊天框 | 任务跑批 | **跨端 Agent OS（PWA/iOS/TG/Web）** |
| 可控 | 不可改 | 不可改 | **公开版可开源 + 私人版完整自用** |

一句话差异化：**别人卖"更强的脑子"，Black God 给你"一个属于你、长在你设备里、越用越懂你的执行中枢"。** 模型是可替换的零件，懂你 + 执行 + 记忆 + 归属才是护城河。

## 4. 系统架构

Black God 一开始就不适合只放在手机本地（iOS 沙盒太受限），所以是**前端轻、服务器重**的分层结构。服务器不是简单转发 API，而是它的**执行脑和工具中枢**。

```text
┌──────────────────────────────────────────────┐
│  前端层（多端入口）                              │
│  PWA · iOS App · Telegram · Web · 浏览器/桌面端  │
├──────────────────────────────────────────────┤
│  Black God Server（执行脑）                      │
│  ┌────────────────────────────────────────┐  │
│  │  核心层                                   │  │
│  │  人格层 · System Prompt · Agent Loop      │  │
│  │  Function Calling · 模型网关 · 多智能体     │  │
│  │  记忆系统 · 技能系统 · 推理 / 路由层         │  │
│  └────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────┐  │
│  │  执行层（沙箱内）                          │  │
│  │  Shell · 文件 · 浏览器 · 搜索 · 部署 · Git  │  │
│  │  图片 / 语音 / 视频 · Cloudflare · iOS 原生 │  │
│  └────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────┐  │
│  │  数据层（本地优先）                        │  │
│  │  长期记忆 · 项目状态 · 用户偏好             │  │
│  │  任务记录 · 技能经验 · 禁碰清单             │  │
│  └────────────────────────────────────────┘  │
├──────────────────────────────────────────────┤
│  模型层（经网关）                                │
│  Claude · DeepSeek · GPT · NVIDIA · 本地模型     │
└──────────────────────────────────────────────┘
```

**执行三态**：本地（隐私强、响应快、轻任务）／ 远端（能力强、可长跑、重任务）／ 混合（**本地懂你，远端干活**）。

## 5. 核心子系统

### 5.1 记忆系统 —— 把经历变成经验

记忆不能无限堆原文，要像人一样**记重点不记废话、记规律不记噪音、记偏好不记流水账、记失败教训避免再犯**。

**压缩管线**：
```text
原始对话/任务 → 提取事实 → 合并重复 → 压缩成摘要
            → 标记重要度 → 写入长期记忆 → 低价值内容自动遗忘
```

**记忆分层（建议的数据结构）**：

| 层 | 内容 | 生命周期 | 例 |
|---|---|---|---|
| 即时 | 当前对话 / 任务上下文 | 单次会话 | 这一步在做什么 |
| 短期 | 最近几天的任务、待办、偏好 | 滚动数天 | 上次部署到哪了 |
| 长期 | 稳定偏好、项目规则、账号结构、**禁碰清单** | 持久 | 哪个服务器不能碰 |
| 技能 | 从任务提炼的可复用流程 | 持久、可淘汰 | Git 工作流套路 |

> 每条长期记忆建议带：`type / value / confidence / source / 重要度 / 最后命中时间`，方便压缩、遗忘和审计。

### 5.2 技能系统 —— 从真实任务里自己长出来

技能不是 App 商城里随便下载的一堆垃圾，而是**用过、验证过、可复用，才配成为技能**。来源三种：①任务沉淀（反复做同类任务 → 总结固定流程）②失败复盘（失败 → 分析 → 把修复沉淀为技能）③高频需求（常出现 → 抽象成可复用能力）。

**质量闸门（拒绝垃圾技能）**：
```text
低频 → 不沉淀     一次性 → 不沉淀     不稳定 → 不沉淀
重复能力 → 合并   过期能力 → 淘汰     高价值 → 强化
```
追求的不是数量，是**命中率高、真实解决问题、不拖慢系统、不互相冲突、不污染人格和记忆**。

> 技能 = Black God 的插件系统，思路来自 Minis / Claude Skills / Agent Skills。已有方向：web-search、project-retrospective、ui-master、ios-collaboration、voice-note-ingest 等。

### 5.3 沙箱与执行 —— 让它能"动手"，但被关在笼子里

没有沙箱，Black God 只是聊天框；有了沙箱，它才是执行助理。沙箱承担：Shell、Python/JS 脚本、文件读写、依赖安装、代码测试、数据处理、临时任务隔离、结果回传。

> **沙箱不只是为了"能做事"，更是为了"安全地做事"**——它同时是执行环境和隔离边界（详见 §6）。

### 5.4 模型网关 —— 你不该关心后面是谁

```text
用户只看 Black God
底层自动路由 Claude / DeepSeek / GPT / NVIDIA / 本地模型
（按任务类型、成本、可用性、是否被拒来选脑子）
```
所以你要的是"Black God 模型"，不是"Claude 壳子"。Key 失效 / 模型拒绝 / 限流 / 成本高时自动切换、降级或重试。

### 5.5 人格层 —— 让 Agent 有稳定风格

人格层不是角色扮演，而是 Black God 的**交互策略**：
```text
懂你 → 预判 → 少问 → 直接做
```
它知道你怎么说话、烦什么、要快、不想听废话、能读懂你没说出口的意思。**公开版用中性助手人格；私人版保留"赵思涵"人格 + 完整偏好（prompt_core / real_dialog_engine）。**

### 5.6 手机 App 商城 —— 把真实 App 接进任务链

先分清两个概念：**App 商城 = 给手机下载真实 App 的入口；技能系统 = Black God 自己推演沉淀的能力。两者不能混。**

App 商城的核心不是陈列应用，而是**和手机端形成端到端闭环**：

```text
识别任务需要某个手机 App → 判断设备系统 + 地区
  → 跳转正确应用市场（App Store / 华为·小米·OPPO·vivo·应用宝 / Google Play / TestFlight·APK·MDM / PWA 添加到主屏）
  → 用户安装或打开 → 完成授权 → Black God 接回上下文 → 继续执行
```

> 本质：**让 Black God 知道手机上该用哪个 App、装没装、没装去哪儿装、装了直接开、要授权就引导**，把真实 App 变成执行链的一部分。

### 5.7 首次安装 —— 先认识你，再服务你

第一次打开先建立**本地画像**：称呼、身份 / 工作方向、常用任务、沟通风格、活跃时间、常用入口、本地偏好。目的只有一个：

> **从第一天起就像认识你，而不是像一个陌生客服。**

## 6. 信任、安全与边界（这一节决定它能不能给别人看）

一个会跑 Shell、改文件、远程执行、做逆向的 Agent，**边界写得越清楚，越值得信任、越敢公开、越容易过审**。2026 年整个 Agent 基建（沙箱化执行、权限最小化、可审计）都在往这个方向收，Black God 应把这三条作为一等公民。

### 6.1 隐私模型：本地优先
```text
本地读取 · 本地保存 · 本地分析 · 不上传云端 · 不交给第三方
```
画像和偏好默认只在本机。**Black God 可以懂你，但不靠偷传数据来懂你**——它是设备里的私人助理，不是云端监控你的人。

### 6.2 权限模型：授权多少懂多少
未授权时进入**默认陌生人模式**：不编造画像、不假装知道你的习惯、不调用未授权数据、只按普通助手响应，等你主动授权再逐步建立画像。
```text
你授权多少，它就懂多少；你不给，它就当陌生人。
```
建议落地为**分级权限 + 显式授权**：读文件、跑命令、访问网络、远程执行、调起手机 App，每一类首次使用都要明确授权，并可随时撤销。

### 6.3 执行安全：沙箱隔离 + 可审计 + 可回滚
- **隔离**：所有执行默认在沙箱内，不直接碰宿主机；远端执行用独立环境 + 最小权限凭证。
- **审计**：每一次工具调用、命令、文件改动都留可追溯日志（谁、何时、改了什么、花了多少），支持回放——对照 LangGraph 的 checkpoint、Claude Code 的 per-tool 成本追踪。
- **危险操作确认**：删除、覆盖、对外发布、动生产服务器等高危动作走二次确认（呼应你现在 `/api/confirm` 的设计）。
- **禁碰清单**：长期记忆里的"不能碰"（生产服务器、私人仓库等）是硬约束，执行层强制校验。

### 6.4 红线：明确"不做什么"
"Black" 代表的是**深层控制力和执行力，不是无法无天**。为了让产品可发布、可信任，建议在文档里就钉死边界——能力只服务于**你自己的、或你获得授权的系统**：
- 逆向 / 安全分析类技能（app-reverse-engineering、security-expert）只用于**自己的设备、自己的应用、或明确授权的目标**（学习、自查、合规安全研究）。
- 不做未授权入侵、不碰他人系统、不批量爬取受保护数据、不协助绕过他人安全机制。
- 远程执行只连**你自己配置的**服务器 / 网关。

> 这一节不是给项目降功率，而是把"暗黑执行力"放进一个**可信任的笼子**里——这正是它从"听起来很猛的玩具"变成"敢自用、敢展示、能长期跑"的产品的分界线。

## 7. 双轨：公开版 vs 私人版

这是项目后期最关键的设计分叉。

| | 公开版（产品外壳） | 私人版（真正灵魂） |
|---|---|---|
| 目标 | 给别人看 / 放 GitHub / 展示能力 | 真正自用 |
| 人格 | 中性助手 | 赵思涵人格 + 完整偏好 |
| 内容 | 正规、中性、不带私人信息 | 完整行为风格、prompt_core、real_dialog_engine |
| 安全 | 不暴露 token / 服务器密码 / 个人关系设定 | 本机保留 |

不分轨：公开仓库会污染隐私；只做公开版：又失去你真正要的 Black God。**所以必须双轨。**

## 8. 命名由来

**Black**：黑客能力、暗黑风格、深层控制、隐蔽执行——不是白净客服机器人，而是"能进（自己授权范围内的）系统、能调工具、能联网、能执行、能替你摆平问题"的智能体。

**God**：能力聚合、多工具调度、全局视角、像一个中枢——不只是模型，而是系统。

所以才有"神"字 Logo：**黑曜石底、冷金属质感的执行中枢**——有控制力，不是可爱助手。

## 9. 现状与差距（诚实自评）

方向是对的，问题在**工程收口**：

1. **概念已经很大，工程还没闭合**：文档写了 Agent OS，但 Push API 没接上、Voice API 有模块没挂路由、`/api/confirm` 前端有调用后端缺、线上还跑旧 v2.2、本地与远程分支曾经混乱。——这是收口问题，不是设计问题。
2. **前端跑在后端前面**：v3.1 前端已有执行时间线、WebPush、本地引导、人格设置、金属 Logo、3 Tab，但后端还是旧路由 → 用户看着像完整产品，点到某些能力却接不上。
3. **能力宣称 ≠ 接口落地**：工具 / 记忆 / 技能 / 多智能体 / 自进化 / 推理层都有，但没全部通过统一 API 暴露给前端。

> **当前关键任务不是再加新概念，而是：把已有能力接线，变成可调用、可测试、可展示。**

## 10. 路线图（按"收口 → 接线 → 扩展"排序）

| 阶段 | 目标 | 关键动作 | 完成标准 |
|---|---|---|---|
| **P0 收口** | 让线上跑起来、不再有"假接口" | 升级线上到 v3+；接上 Push / Voice 路由；补 `/api/confirm` 后端；统一分支 | 前端每个按钮后端都接得上 |
| **P1 接线** | 已有能力统一暴露 | 工具 / 记忆 / 技能 / 模型网关走统一 API；可调用、可测试 | 一条任务能走完整 Agent Loop 并落审计日志 |
| **P2 安全** | 让它敢自用敢展示 | 落地权限分级、沙箱隔离、审计日志、禁碰清单校验、危险操作确认 | 通过一遍自己的"红线"清单 |
| **P3 记忆/技能闭环** | 越用越省、越用越懂 | 压缩管线 + 遗忘 + 技能质量闸门跑通 | 单位任务模型成本随时间下降 |
| **P4 扩展** | 多端 + 端到端 | App 商城端到端链路、Telegram、iOS 原生能力、多智能体 | 手机真实 App 进任务链 |

## 11. 成功标准（怎么知道它真的成了）

不靠"功能多"判断，靠这几条硬指标：

- **省 Key**：单位任务的模型调用成本**随使用时间下降**（本地 / 缓存 / 工具命中率上升）。
- **懂你**：重复类任务里"主动接住上下文、少问、直接做"的比例上升；需要你重新解释背景的次数下降。
- **会做**：端到端任务（理解 → 执行 → 验证 → 交付）成功率，而不是"给建议"的比例。
- **可信**：每次危险操作都有确认 + 审计；零越界执行；画像零意外上云。
- **长大**：技能库命中率维持高位的同时，垃圾技能被自动淘汰（数量不膨胀、质量在升）。

## 12. 风险登记（简版）

| 风险 | 影响 | 缓解 |
|---|---|---|
| 概念过大、工程跟不上 | 永远像 demo | 严格按 P0→P4 收口，先接线再扩展 |
| 执行能力被误用 / 越界 | 安全与合规风险 | §6 红线 + 权限分级 + 禁碰清单 + 审计 |
| 强品牌叙事吓退用户 / 审核 | 难展示、难过审 | 公开版收敛措辞，边界写在明处 |
| 单点服务器 | 远端挂了全瘫 | 本地降级模式 + 远端可替换 |
| 记忆膨胀 / 隐私泄露 | 又慢又不安全 | 压缩 + 遗忘 + 本地优先 + 加密存储 |
| 模型全家桶都不可用 | 服务中断 | 模型网关多路由 + 本地小模型兜底 |
