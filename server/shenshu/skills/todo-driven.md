# 神枢 Agent 执行技能

## 核心原则

复杂任务(预估执行步骤 > 5)必须先落盘 todo.md，再开始执行。
简单任务(1-4步)直接执行，不需要这套机制。

## 触发条件

用户提出的任务满足以下任一:
- 明确要求"分步完成"、"做一个完整的XXX"
- 预估需要调用 5次以上工具
- 涉及多个独立模块(如: 造应用+部署+推送)
- 用户主动要求"列个清单"

## 执行流程

### 1. 生成 todo.md

写入 `/var/minis/workspace/todo_<任务简称>.md`:

```markdown
# 任务: <一句话描述用户目标>

## 状态: 进行中
## 创建时间: <timestamp>

## 步骤清单
- [ ] 1. <具体可验证的子任务>
- [ ] 2. <具体可验证的子任务>
- [ ] 3. <具体可验证的子任务>
...

## 上下文锚点
- 关键约束: <用户明确提出的限制条件>
- 关键决策: <执行过程中做出的重要选择,后续步骤要遵守>
- 已知坑: <踩过的坑,避免重复踩>
```

### 2. 逐步执行

每完成一个子任务:
1. 立即用 file_edit 把对应 `[ ]` 改成 `[x]`
2. 如果发现新的必要步骤，追加到清单末尾
3. 如果某步骤失败，在该行后面加注 `⚠️ 失败原因: xxx`，不要跳过不记录

### 3. 上下文守恒检查点

以下情况必须重新读取 todo.md:
- 单次对话轮次超过 15 轮
- 连续调用工具超过 8 次
- 中途被用户打断去做别的事，之后要继续时
- 感觉"有点忘了最初要做什么"的任何时刻

### 4. 交付检查

任务收尾前，必读一遍 todo.md 全文，确认:
- [ ] 所有清单项都已勾选或明确注明放弃原因
- [ ] 用户最初的目标(标题那句话)已经达成
- [ ] 没有半途而废的临时文件/未清理的中间产物

### 5. 收尾

任务完成后:
1. todo.md 顶部状态改为 `已完成`
2. 用一句话总结交付了什么
3. 不需要主动删除 todo.md，留作可追溯记录

## 与神枢分档路由的配合

todo.md 里每个步骤可以标注建议复杂度，供神枢网关路由参考:

```markdown
- [ ] 3. 重构核心模块架构 [complexity: high]
- [ ] 4. 修正文档里的错别字 [complexity: low]
```

复杂度标注不是强制的，但标注了神枢会优先按标注路由模型，没标注则按默认规则(内容长度+关键词)自动评估。

## 反面案例(禁止行为)

- ❌ 收到复杂任务直接开始执行，不写 todo.md
- ❌ 写了 todo.md 但执行完不更新勾选状态
- ❌ 半途发现任务范围变了，不更新清单就自己瞎改目标
- ❌ 遇到失败步骤直接跳过，不记录原因
